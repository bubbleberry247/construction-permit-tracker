# メール分類ルール（形式知）

shinsei.tic@gmail.com に届くメールの対象/対象外判定ルール。
2026-03-26時点で79件のメールを分析して策定。

---

## 分類カテゴリ

| カテゴリ | 対応 | 件数 |
|---------|------|------|
| A. 業者提出（PDF添付） | OCRパイプライン処理 | 51 |
| B. 業者提出（ZIP添付） | ZIP展開→OCR処理 | 9 |
| C. 業者提出（XLSX記入済のみ） | 書類不足→再提出依頼 | 4 |
| D. 業者提出（JPG添付） | JPG→PDF変換→OCR処理 | 0 |

| G. 業者からの質問 | 担当者対応フラグ | 2 |

| I. 自社テンプレート送信 | 対象外（東海インプル→業者への送信） | 0 |

## 判定ルール（優先順）

### ルール1: システム通知 → 対象外
**条件（いずれか）:**
- 元送信者が `no-reply@accounts.google.com` or `mailer-daemon@googlemail.com`
- 件名に「セキュリティ通知」「重大なセキュリティ通知」
- 件名に「Delivery Status Notification」
- 件名に「ビジネスに関するヒント」

**根拠:** Google/Gmailのシステム通知。業者提出とは無関係。

---

### ルール2: クラウドDL必要 → 要ダウンロード処理
**条件:**
- 本文に「添付ファイル分離」「添付ファイルは予め分離」を含む
- 本文にダウンロードURL（activegate, digitalartscloud等）を含む
- 本文に「ダウンロードセンター」を含む

**特徴:**
- 別メールでパスワードが送られてくる（ルール3と対応）
- 業者のメールシステムが自動的に添付を分離する仕組み
- ファイルの実体はクラウド上にあり、メール自体にはPDF添付なし

**対応フロー:**
1. メール本文からダウンロードURLを抽出
2. 対応するパスワード通知メール（ルール3）を件名・送信者で照合
3. Playwrightでダウンロードセンターにアクセス
4. メアド入力 → パスワード入力（or OTP受信）→ ファイルDL
5. DLしたファイルを添付して **shinsei.tic@gmail.com から自分宛に転送メールを送信**
6. 転送メールがkalimistkに転送される → fetch_gmail.pyで通常処理

**重要制約:**
- 転送メールの送信元は **shinsei.tic@gmail.com のみ**。kalimistk@gmail.comからは送信しない
- 理由: shinsei.ticが正式な受信メールアドレスであり、業者とのやり取りの記録をshinsei.tic側に残す必要がある
- 実装: Gmail APIでshinsei.ticとして送信（OAuth2認証が必要）、または手動でshinsei.ticから転送

**実例:**
- ActiveGate（s-thing.co.jp / 高橋由衣）: URL + パスワード別メール
- Digital Arts Cloud（ota-shoji.co.jp / 太田商事）: URL + OTPメール

---

### ルール3: パスワード通知 → ペアメール（単体処理不要）
**条件（いずれか）:**
- 件名に「[PASSWORD]」「パスワード通知」
- 本文に「パスワードをお知らせします」「Password is as follows」
- 本文に「暗号化後のファイル名:」（ZIPパスワード）
- 送信者が DigitalArts Cloud N.（OTP通知）

**対応:**
- ルール2（クラウドDL）またはルールB（ZIP展開）のパスワードとして使用
- 件名・送信者・日時で対応するメールを照合

---

### ルール4: 業者からの質問 → 担当者対応フラグ
**条件:**
- 本文に質問表現: 「必要でしょうか」「でしょうか？」「ですか？」
- 添付ファイルなし or 少量
- 元送信者が業者ドメイン

**実例:**
- インフラテック冨田: 「工事（施工）の登録書のようですが、弊社でも必要でしょうか？」
- 総合建装: 質疑回答の送付

**対応:** 東海インプル建設の担当者に転送・回答依頼

---

### ルール5: 業者提出（PDF直接添付）→ OCR処理
**条件:**
- 添付に .pdf ファイルあり（サイズ > 1KB）
- ルール1-4に該当しない

**特徴:**
- 最も一般的なパターン（51/79件 = 65%）
- 1通に複数PDF添付のケースあり（決算書+許可証+申請書等）
- 転送メール（Fwd:）の場合、元送信者が業者

---

### ルール6: 業者提出（ZIP添付）→ 展開→OCR処理
**条件:**
- 添付に .zip ファイルあり
- ルール1-4に該当しない

**特徴:**
- パスワード付きの場合あり（ルール3のパスワードメールと照合）
- ZIP内に複数PDF（書類一式）が含まれる
- ファイル名がShift-JISエンコードの場合あり（cp437→shift_jis変換必要）

---

### ルール7: 業者提出（XLSXのみ）→ 書類不足
**条件:**
- 添付に .xlsx のみ（PDF/ZIPなし）
- 元送信者が業者ドメイン（shinsei.ticではない）

**判断:**
- Excelテンプレートの記入済みファイルのみ送付
- 建設業許可証等のPDF添付を忘れている可能性が高い
- → 再提出依頼が必要

**注意:** 東海インプル自身がテンプレートを送った転送メールとの区別が必要
- 自社送信: 件名に「訂正版送付」+ 元送信者がshinsei.tic
- 業者返送: 元送信者が業者ドメイン

---

### ルール8: 業者提出（JPG/PNG添付）→ PDF変換→OCR処理
**条件:**
- 添付に .jpg/.jpeg/.png あり（サイズ > 10KB）
- icon/image00x等のインライン画像を除外
- PDF添付なし

**実例:**
- 建設業許可.jpg（許可証のスキャン画像）

---

### ルール9: OTP通知 → ペアメール
**条件:**
- 送信者が「DigitalArts Cloud N.」
- 件名に「f-FILTER ワンタイムパスワード」

**対応:** ルール2のDigital Arts CloudダウンロードのOTPとして使用

## 重複・訂正版の判定ルール

### 同一会社の複数メール
- **同一ファイルハッシュ**: 重複（TIER1_DUPLICATE）→ スキップ
- **件名に「訂正版」「再送」**: 最新メールのファイルを採用
- **件名に「添付漏れ分」**: 前回メールの添付に追加（差分統合）
- **ファイル名が同じで内容異なる**: 日付が新しい方を採用

### 判定に使えるシグナル
| シグナル | 意味 |
|---------|------|
| 件名に「訂正版送付」 | 前回の不備を修正した再送 |
| 件名に「再送」「お取引条件等説明書を再送します」 | 同内容の再送 |
| 件名に「添付漏れ分」 | 前回メールへの追加添付 |
| 件名に「Re:」+ 業者ドメイン | 業者からの返信（書類 or 質問） |
| 件名に「Fwd:」+ shinsei.tic | shinsei.ticからの転送 |
| 同一送信者 + 同日 | 同一業者の追加送付の可能性 |

## 元送信者の特定方法

shinsei.tic → kalimistk の転送メールでは、From: は shinsei.tic になる。
元の送信者はメール本文の「Forwarded message」セクションから取得:

```
---------- Forwarded message ---------
From: 石川正之 <goemon1230@gmail.com>
Date: 2026年3月19日(水) 9:57
Subject: ...
To: shinsei.tic@gmail.com
```

パターン: `From:` + メールアドレスを含む行を本文から正規表現で抽出

## 未分類・例外ケース

1. **CSV添付のみ**: 取引先一覧表をCSVで送付（Excel→CSVエクスポート）
2. **PNG添付（インライン）**: メール署名のロゴ画像。image001.png等は除外
3. **GIF添付**: メールテンプレートの装飾。icon.png等は除外
4. **複合添付**: PDF + XLSX + CSV + JPG が混在する場合あり
